Terser code is rarely simpler

@gc_psk / July 14, 2020

This is a post that old me (2-5 years ago) would have disliked (is this a sign of maturity?). Also: it's a bit of a rant agains overly-terse code.

When Babel came out and let us enjoy ES6's features, my Javascript style changed into a much terser form.

Arrow functions let use write complex functions in a single-line, ternary operators just became cool again, and if statements were too procedural, I'd have said.

Problems with terse code

While there is an argument to be made about the elegance of terse code, elegance is not a synonym of simplicity. Most often than not, terser code is rarely simpler. And this is not to be confused with less code.

Instances where I don't prefer terse code:

  • logical operators to replace conditional statements
  • long or complex one-liners

If statements

This is not an ode to if statements. We all write conditional branching, in a way or another. Sometimes, it's really helpful and elegant to write conditional branching using a logical operator: for example, for variable assignment. In many cases, that's fine, especially if the right-hand assignment is short and simple.

The point is: don't be scared of if statements. If they read better than a contorted ternary operator, or logical assignment, just use them.

One-liners

One-liners give me a headache, especially when I have to refactor or modify them.

They are harder to read, cause diff-noise if I have to add a new line, cause longer horizontal lines. And I seriously dislike reading horizontally.

I see the point of using them in some cases, but I see them used and abuse too often.
